# 제품 루프와 게임 경제

상태: **설계 기준선 / 수치 provisional**

## 플레이어 선택

NFT 획득 후 최소 세 가지 선택이 있어야 한다.

1. 장착해 현재 build를 강화한다.
2. 마켓에 판매해 AQT를 얻는다.
3. 분해해 클래스 성장과 제작에 필요한 재료를 얻는다.

클래스 NFT는 네 번째 선택인 새 플레이 스타일을 제공한다. 전직은 구매만으로
완료되지 않고 클래스 숙련도, 지정 스테이지 클리어와 재료 조건을 함께 요구한다.

장비 보너스는 스테이지를 클리어할 때마다 누적되지 않는다. wallet은 여러 NFT를
보유할 수 있지만 전투에는 Weapon, Armor, Accessory 각 하나만 적용한다. 장착은
귀속이나 소각을 만들지 않으며, 판매하려면 먼저 장착을 해제한다. 중복 장비는
판매 또는 향후 분해/제작의 공급원이 된다.

서로 다른 클래스 NFT는 wallet의 선택 가능한 클래스 목록으로 누적한다. 같은
클래스 NFT를 여러 개 보유해도 능력치, 스킬 또는 entitlement가 배수로 중첩되지는
않는다. 중복 클래스 NFT는 거래 또는 향후 전직 재료 정책의 대상이다.

## 40개 스테이지 진행 밴드

| 구간 | 진행 목적 | 보상 방향 |
| --- | --- | --- |
| 1–10 | 기본 조작과 첫 build | 일반/고급 장비, 기본 클래스 조각 |
| 11–20 | 3슬롯 build 형성 | 희귀 장비, 첫 전직 준비 재료 |
| 21–30 | 전직 build 확장 | 영웅 장비, 전직 affix와 제작 재료 |
| 31–40 | 고난도 완성 | 전설 후보, 고유 boss affix와 종결 recipe |

10, 20, 30, 40은 확정 보상 또는 pity 진전을 주는 경계 boss로 취급한다.
완성 클래스 NFT의 낮은 확률 drop은 허용할 수 있지만, 조각을 모으면 반드시
획득 가능한 경로를 함께 둔다.

## 수직 성장과 수평 성장

- 수직 성장: 체력, 피해, 방어와 쿨다운의 제한된 증가
- 수평 성장: 이동형/보스형/생존형 affix, 클래스 경로와 스킬 선택

필수 스테이지는 NFT 미장착 기본 클래스의 합리적 숙련도로 클리어 가능해야 한다.
NFT는 시간 단축과 build 다양성을 주되 입장권이 되어서는 안 된다.

후반 장비가 초반 장비의 모든 수치를 항상 초과하면 초반 NFT 시장은 사라진다.
따라서 낮은 tier에도 다음 중 하나를 남긴다.

- 특정 스킬이나 이동 동작과 결합하는 고유 affix
- 높은 roll이 가능한 좁은 전문 능력치
- 유용한 분해 산출량 또는 recipe 입력
- cosmetic variant와 수집 이력

## 임시 전역 상한

아래 값은 resolver와 플레이 테스트를 시작하기 위한 provisional 값이다.

| 능력치 | 합산 상한 |
| --- | ---: |
| 스킬 피해 증가 | 2,500 bps |
| 쿨다운 감소 | 2,000 bps |
| 이동속도 증가 | 1,500 bps |
| 최종 피해 감소 | 2,500 bps |
| 최대 체력 증가 | 3,000 bps |

상한과 적용 순서는 `balanceVersion`별 순수 함수로 관리한다. UI 문자열이나
metadata JSON을 파싱해 전투 수치를 계산하지 않는다.

표의 쿨다운 2,000 bps는 equipment source 자체의 상한이다. class/skill
enhancement와 합산한 최종 cooldown reduction은 현재 4,000 bps를 상한으로
사용해 max-enhanced class에서도 장착 NFT 효과가 사라지지 않게 한다.

## 공급과 소각

발행:

- 검증된 stage reward
- 조각 recipe를 통한 클래스 NFT 제작
- 이벤트/업적은 별도 제품 결정 이후

소각:

- 장비 NFT 분해
- ERC-1155 조각/정수의 클래스 제작과 전직 소비
- 향후 합성 recipe

초기에는 내구도, 수리 비용, 무작위 강화 실패와 파괴를 도입하지 않는다.
이들은 자산 소유 경험을 해치고 경제 튜닝 전에는 공급 문제를 가릴 수 있다.

## 공정한 RNG

- 브라우저 `Math.random`과 클라이언트 제출 rarity를 사용하지 않는다.
- 서버가 검증 완료 후 versioned drop table로 결과를 한 번 계산하고 원자적으로 저장한다.
- 같은 attempt 재시도는 항상 같은 저장 결과를 반환한다.
- attempt 포기 반복을 제한하기 위해 만료, 동시 attempt 제한과 rate limit을 둔다.
- pity 카운터는 wallet, drop pool과 balance version에 묶고 DB 경합에서도 한 번만 증가한다.

## 관측 지표

- stage별 시도/클리어/포기율
- rarity와 affix별 발행량
- 장착률, 거래율, 분해율과 장기 미사용률
- 클래스별 선택률, 클리어 시간과 사망률
- 슬롯별 능력치 분포와 상한 도달률
- 마켓 등록 대비 체결률과 가격 분산
- wallet별 pity 진행과 확정 획득 시간

지표는 개인을 추적하기 위한 것이 아니라 공급·밸런스와 막힌 진행을 찾기 위한
집계값으로 사용한다.

## 경제 인수 조건

- 기본 장비만으로 모든 필수 stage를 이론상 완료할 수 있다.
- 한 능력치만 쌓는 build가 모든 역할에서 최적이 아니다.
- 낮은 tier NFT의 장착·분해·거래 중 하나의 사용률이 관측된다.
- 모든 확률 보상에는 버전, 확정 경로와 중복 방지가 있다.
- 발행/소각 수치 변경은 새 balance version과 rollback/containment 기록을 요구한다.
