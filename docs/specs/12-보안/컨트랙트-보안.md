# 컨트랙트 보안 검토

상태: **기준선**
담당: **보안 에이전트**
상위 문서: [보안 명세](README.md)

## 목적

권한·재진입·DoS·토큰·선행매매를 검토한다. 새 구현에는 전부 적용하고, 기존 구현은 관련 코드를 수정할 때 이
기준으로 점진적으로 이전한다.

## 구현 계약

- 입력과 외부 데이터는 타입과 런타임 스키마로 검증하고 크기를 제한한다.
- 상태는 한 소유자만 변경하며, 좁은 인터페이스와 명시적인 실패 타입을 쓴다.
- ID·단위·시간 기준·버전·chain ID·주소·반올림 규칙을 명시한다.
- 도메인 간 통신은 직렬화 가능한 타입/이벤트만 사용하고 Phaser 객체,
  DB client, wallet client 또는 비밀을 경계 밖으로 전달하지 않는다.
- 재시도·동시 요청·중단·씬 종료·부분 실패 후에도 일관성이 유지되어야 한다.

## 실패 및 보안 규칙

- 누락·만료·중복·오소유자·오네트워크·지원하지 않는 버전·범위 초과를
  되돌릴 수 없는 효과 전에 거부한다.
- 브라우저와 게임 결과는 보상액·mint·소유권·관리자 작업을 승인할 수 없다.
- 성공은 권위 있는 상태 전이가 끝난 뒤에만 표시하고 안전한 재시도 경로를 둔다.
- 비밀·cookie·private key·seed phrase·원본 인증 서명을 코드와 로그에 남기지 않는다.
- 메인넷·자금·관리자 권한·운영 데이터 파괴는 사람 승인 패킷을 요구한다.

## 인수 조건

- [ ] 관련 구현 파일, 타입, 이벤트/API/ABI와 책임자가 연결되어 있다.
- [ ] 정상·경계값·실패·재시도/경합·정리 테스트가 존재한다.
- [ ] 권한과 보안 불변조건을 필요한 경우 독립적으로 검토했다.
- [ ] lint, typecheck, 집중 테스트와 위험에 따른 build/E2E/컨트랙트 검사가 통과한다.
- [ ] 관측성, rollback 또는 불변 컨트랙트의 봉쇄 절차가 최신이다.

## 프로젝트 결정 기록

구현 시 실제 경로·수치·다이어그램·예외·테스트 증거를 여기에 추가한다.
예외에는 책임자, 이유, 보완 통제와 재검토 날짜가 반드시 있어야 한다.
